# 設計 → 実装

## 参照する前提

- 常に @doc/requirements.md を読み、技術スタック/非機能要件/用語を把握する。
- 設計書は $ARGUMENTS を指定があればそれを、なければ @doc/design.md を読む。
- 実装前提が曖昧な場合は最小限の質問で確定する（権限境界、データ保持、通知条件、オフライン挙動）。

## 手順

1. 設計書から実装対象（機能・画面・データ・非機能）を抽出する。
2. 実装単位に分解し、依存関係と実装順序を決める。
3. Firestore/Storage/FCM の具体設計（コレクション・ルール・トリガー）をタスク化する。
4. 画面ごとの UI 構成と状態管理の粒度を決める（画面状態、フォーム、一覧、詳細）。
5. 権限/セキュリティ/退会時の削除をルールと実装に反映する。
6. 非機能要件に対する実装上の注意点を明記する（性能/アクセシビリティ/オフライン）。
7. TDD 前提でテスト観点とモック方針を整理する。

## 出力

- @.claude/skills/references/implementation-template.md をテンプレートとして埋める。
- 日本語で、設計書の用語に合わせる。
- 仮定と未確定事項は最後にまとめる。
